## 交易数据结构

### 核心实体

```
Transaction（交易）
├── id              UUID
├── date            交易日期
├── description     交易描述
│
├── category_id     分类 [可选]
│
├── is_reconciled   核对状态 [默认 false]
│                   -- 预留字段，当前不实现切换功能
│
├── deleted_at      软删除时间戳 [可选]
│                   -- 预留字段，当前采用硬删除策略
│
├── postings        分录列表（至少2条）
│   ├── Posting 1
│   │   ├── account_id  账户
│   │   ├── amount      金额（正=借方，负=贷方）
│   │   └── sequence    分录顺序（用于显示排序）
│   ├── Posting 2
│   │   ├── account_id  账户
│   │   ├── amount      金额
│   │   └── sequence    分录顺序
│   └── ... 更多分录（可选）
│
├── created_at      创建时间
└── updated_at      更新时间

> **设计说明**: 本系统采用纯自由复式记账，不存储交易类型标识。所有交易由用户手动指定 Posting 组合，或通过快速记账 UI 推导 Posting，但数据库层面仅存储 postings，不存储类型元数据。
```

### 新增字段说明

| 字段              | 类型          | 说明                                       |
| ----------------- | ------------- | ------------------------------------------ |
| **is_reconciled** | INTEGER (0/1) | 核对状态：保留字段，当前不实现切换功能     |
| **deleted_at**    | TEXT (可选)   | 软删除时间戳：保留字段，当前采用硬删除策略 |
| **sequence**      | INTEGER       | 分录顺序：用于控制显示顺序（借方在前等）   |

### 删除交易设计

> **设计决策**: 本系统采用**硬删除**策略，用户确认后直接从数据库删除交易记录。

```
删除交易流程：
├── 用户点击删除 → 弹出确认对话框（显示交易信息）
├── 用户确认 → 执行硬删除
│   ├── DELETE FROM postings WHERE transaction_id = ?
│   ├── DELETE FROM transactions WHERE id = ?
│   └── CASCADE 自动清理关联数据
└── 不可恢复
```

**硬删除设计理由**：

- 桌面应用场景下，用户删除操作多为有意识的清理
- 避免软删除数据积累导致查询性能下降
- 简化实现，无需维护"已删除"视图和恢复流程
- `deleted_at` 字段保留用于未来审计需求，当前不使用

### Reconciliation（核对）字段说明

> **设计决策**: `is_reconciled` 字段保留于 schema 和 model，但**不实现核对流程功能**。

```
字段定义：
├── is_reconciled INTEGER NOT NULL DEFAULT 0
├── 标记交易是否已与外部记录核对
└── 当前仅为预留字段，无业务逻辑依赖
```

**保留理由**：

- 未来可能需要核对功能时无需修改 schema
- 字段不影响当前系统运行
- 可用于数据导入时的标记用途

### 创建交易输入要素

| 字段            | 必填性  | 说明                  | 来源                              |
| --------------- | ------- | --------------------- | --------------------------------- |
| **date**        | ✅ 必填 | 交易发生日期          | 用户输入，默认今天                |
| **description** | ✅ 必填 | 描述/备注             | 用户输入                          |
| **postings**    | ✅ 必填 | 至少2条分录，借贷平衡 | 用户选择账户、输入金额            |

## 创建交易流程模式

### 模式A：自由复式记账（专业模式）

```
用户操作流程：
1. 选择日期
2. 输入描述
3. 添加分录（至少2条）
   ├── 选择账户 A，输入金额 +100（借方）
   ├── 选择账户 B，输入金额 -100（贷方）
   └── 可继续添加更多分录
4. 系统验证：sum(amounts) == 0
5. 可选：设置分类
6. 提交创建

适用场景：
- 复杂交易（多账户、多分录）
- 专业用户
- 用户完全控制 Posting 组合
```

### 模式B：快速记账（简化模式）

```
用户操作流程：
1. 选择记账模式（支出/收入/转账）
2. 输入金额
3. 选择来源账户（可选，有默认）
4. 选择目标账户/分类
5. 输入描述（可选）
6. 提交创建

系统自动推导分录（仅UI层，不存储类型）：
├── 支出模式: 资产账户 -金额 → 支出账户 +金额
├── 收入模式: 收入账户 -金额 → 资产账户 +金额
└── 转账模式: 转出账户 -金额 → 转入账户 +金额

> **重要**: 快速记账仅简化用户输入流程，系统根据用户选择的账户类型组合推导 Posting，
> 最终提交到数据库的仍是纯 Posting 列表，不存储"支出/收入/转账"类型标识。

适用场景：
- 日常收支记录
- 普通用户
- 简化的2分录交易
```

### 模式C：账户转账（已实现）

```
现有实现：account_transfer(from_id, to_id, amount, description)
├── 自动创建2分录交易
├── 无分类
└── 分录：转出账户 -amount → 转入账户 +amount

适用场景：
- 资产账户间转移
- 已实现，可复用逻辑
```

### 模式D：期初余额（已实现）

```
现有实现：account_create(..., initial_balance, initial_balance_date)
├── 自动创建 Opening Balance 交易
├── 分录：权益账户 -金额 → 资产/负债账户 +金额
└── 系统自动，用户不直接操作

适用场景：
- 新账户初始化
- 已实现
```

## 快速记账账户类型组合建议

> **说明**: 本系统不存储交易类型标识。以下账户类型组合仅用于快速记账 UI 层推导 Posting，
> 后端不强制验证。用户可通过自由复式记账模式指定任意 Posting 组合。

### 常见记账模式及其 Posting 推导

| 模式     | 用户视角         | 推导规则（借方+，贷方-）                           |
| -------- | ---------------- | -------------------------------------------------- |
| **支出** | 从资产账户花钱   | 资产账户 -amount（贷方），支出账户 +amount（借方） |
| **收入** | 收款到资产账户   | 资产账户 +amount（借方），收入账户 -amount（贷方） |
| **转账** | 资产账户间转移   | 转出账户 -amount（贷方），转入账户 +amount（借方） |
| **期初** | 设置账户初始余额 | 权益账户 -amount（贷方），目标账户 +amount（借方） |

### 账户类型组合参考（UI 层默认选择）

```typescript
// 快速记账 UI 层默认账户选择逻辑（仅用于简化输入，非后端验证）
const QUICK_ADD_DEFAULTS = {
  // 支出：默认从资产账户（贷方）到支出账户（借方）
  expense: {
    credit: ["asset"], // 资金来源账户（贷方，负金额）
    debit: ["expense"], // 支出目标账户（借方，正金额）
  },
  // 收入：默认从收入账户（贷方）到资产账户（借方）
  income: {
    credit: ["income"], // 收入来源账户（贷方，负金额）
    debit: ["asset"], // 资金去向账户（借方，正金额）
  },
  // 转账：资产/负债账户间转移
  transfer: {
    credit: ["asset", "liability"], // 转出账户（贷方，负金额）
    debit: ["asset", "liability"], // 转入账户（借方，正金额）
  },
};
```

> **设计说明**:
>
> 1. 以上组合仅为 UI 层默认账户选择逻辑，后端不存储交易类型，也不验证账户组合。
> 2. 用户可通过自由复式记账模式创建任意 Posting 组合，系统仅验证借贷平衡（sum == 0）。

## 事务与并发控制

### SQLite 事务隔离级别

SQLite 默认使用 **SERIALIZABLE** 隔离级别，保证：

- 读操作看到的一致快照
- 写操作完全串行化
- 无脏读、无不可重复读、无幻读

> **重要**: SQLite 的 SERIALIZABLE 实现通过数据库级锁实现，高并发写入时可能阻塞。

### 交易创建事务边界

```
create_transaction(conn, input)
├── BEGIN TRANSACTION                    -- 开启事务
│   ├── validate_accounts()              -- 账户验证（读取）
│   ├── validate_category()              -- 分类验证（读取）
│   ├── INSERT transactions              -- 插入交易
│   ├── INSERT postings (N条)            -- 插入分录
├── COMMIT                                -- 提交（成功）或 ROLLBACK（失败）
```

**事务保证**:

- 交易、分录原子写入
- 失败时自动回滚，不留孤儿数据
- 借贷平衡验证在事务内完成

### 并发场景与竞态条件

| 场景                 | 风险             | 解决方案                             |
| -------------------- | ---------------- | ------------------------------------ |
| 同时创建相同账户交易 | 余额计算不一致   | 余额由后端计算，前端不缓存           |
| 编辑时删除原交易     | 更新找不到记录   | 事务内先检查 EXISTS，不存在则报错    |
| 分类删除时创建交易   | 分类验证失效     | 事务内验证分类存在（或使用外键约束） |
| 账户关闭时创建交易   | 激活状态验证失效 | 事务内验证 is_active（乐观锁）       |

> **注**: SQLite 单写者模式限制了并发写入吞吐量，但简化了并发控制。桌面应用场景下，单用户操作无真正并发，竞态条件主要来自 UI 快速操作（如双击保存）。

### 乐观锁与悲观锁选择

| 操作         | 锁策略 | 理由                               |
| ------------ | ------ | ---------------------------------- |
| 创建交易     | 无锁   | 新记录无冲突                       |
| 编辑交易     | 乐观锁 | 检查 updated_at 时间戳（可选增强） |
| 删除交易     | 无锁   | CASCADE 自动清理关联               |
| 账户余额更新 | 无锁   | 由交易分录计算，不存储余额字段     |
